# Nion Orchestration Engine

A simplified implementation of Nion's AI orchestration system — an AI Program Manager agent that tracks action items, risks, issues, and decisions across projects using a three-tier **L1 → L2 → L3** agent hierarchy powered by Google Gemini via LangChain.

---

## Architecture

```
Input Message (JSON)
        │
        ▼
┌───────────────────┐
│   L1 Orchestrator │  ← Ingests message, reasons about intent, creates task plan
│  (Google Gemini)  │    Visibility: L2 domains + Cross-Cutting agents only
└────────┬──────────┘
         │ delegates
    ┌────┴────────────────────────────────────┐
    │                                         │
    ▼                                         ▼
L2 Domain Coordinator             L3 Cross-Cutting Agent
(TRACKING_EXECUTION /             (knowledge_retrieval /
 COMMUNICATION_COLLABORATION /     evaluation)
 LEARNING_IMPROVEMENT)
    │
    ▼
L3 Domain Agents
(action_item_extraction, risk_extraction,
 qna, message_delivery, etc.)
```

### Three-Tier Hierarchy

| Layer | Role | Visibility |
|-------|------|------------|
| **L1 Orchestrator** | Ingests message, reasons about intent, generates task plan | L2 domains + Cross-Cutting agents only |
| **L2 Coordinator** | Receives directions from L1, coordinates L3 agents, aggregates results | Its own L3 agents only |
| **L3 Agent** | Executes specific tasks, returns structured results | None (leaf node) |

### Available Agents

**L2 Domains (visible to L1):**
- `TRACKING_EXECUTION` — extraction and tracking of action items, risks, issues, decisions
- `COMMUNICATION_COLLABORATION` — Q&A, reporting, message delivery, meeting attendance
- `LEARNING_IMPROVEMENT` — learning from explicit instructions and SOPs

**Cross-Cutting Agents (visible to L1 and all L2s):**
- `knowledge_retrieval` — retrieves project context, stakeholder details, historical data
- `evaluation` — validates outputs before delivery

**L3 Agents under TRACKING_EXECUTION:**
`action_item_extraction`, `action_item_validation`, `action_item_tracking`,
`risk_extraction`, `risk_tracking`, `issue_extraction`, `issue_tracking`,
`decision_extraction`, `decision_tracking`

**L3 Agents under COMMUNICATION_COLLABORATION:**
`qna`, `report_generation`, `message_delivery`, `meeting_attendance`

**L3 Agents under LEARNING_IMPROVEMENT:**
`instruction_led_learning`

## API Key & Quota Guide (Read Before Running)

This project uses the **Google Gemini API** free tier. Understanding quota limits will save you frustration.

### Free Tier Limits (2025)

| Model | Requests/Day | Requests/Min | Recommended For |
|-------|-------------|-------------|-----------------|
| `gemini-2.0-flash` | 1500 | 15 | ✅ Running all test cases |
| `gemini-1.5-flash` | Deprecated | — | ❌ Do not use — returns 404 |
| `gemini-2.5-flash` | 20–25 | low | ❌ Too low for full test suite |



### Each test case makes ~8–12 API calls
Running all 7 test cases uses roughly 70–80 calls total. With the free tier you can comfortably run the full suite — but only if you pace the calls (the engine does this automatically with a 15-second delay between test cases).

### What to do if you hit quota limits

**Option 1 — Wait and retry (free)**
- Per-minute limit (15 req/min): wait 60 seconds and retry
- Per-day limit: resets at midnight Pacific Time
- Run test cases one at a time: `python main.py --test 2 --save`

**Option 2 — Create a new API key (free)**
You can create multiple free API keys on different Google projects:
1. Go to [https://aistudio.google.com/app/apikey](https://aistudio.google.com/app/apikey)
2. Click **"Create API key in new project"** (not an existing project)
3. Replace `GOOGLE_API_KEY` in your `.env` with the new key
4. Each new project gets its own fresh quota

**Option 3 — Enable billing (paid, negligible cost)**
If you need to run the full suite without any limits:
1. Go to [https://console.cloud.google.com/billing](https://console.cloud.google.com/billing)
2. Link a billing account to your Google Cloud project
3. `gemini-2.0-flash` costs ~$0.075 per 1M tokens — running all 7 test cases costs less than $0.01 total

> **Note for evaluators:** A valid free-tier key with `gemini-2.0-flash` is fully sufficient to run and verify this project. Run one test case at a time to stay within the 15 req/min rate limit.

## Setup — Complete Guide

### Prerequisites
- Python 3.9+
- Git

### Step 1: Clone the Repository

```bash
git clone https://github.com/Manoj-1918/Nion-Orchestration.git
cd Nion-Orchestration
```

### Step 2: Create a Virtual Environment

```bash
# Windows
python -m venv venv
venv\Scripts\activate

# macOS / Linux
python3 -m venv venv
source venv/bin/activate
```

### Step 3: Install Dependencies

```bash
pip install -r requirements.txt
```

### Step 4: Get a Free API Key

1. Go to [https://aistudio.google.com/app/apikey](https://aistudio.google.com/app/apikey)
2. Sign in with your Google account
3. Click **"Create API key in new project"**
4. Copy the key — it starts with `AIza` and is ~39 characters long

### Step 5: Configure Your Environment

```bash
cp .env.example .env
```

Open `.env` and add your key:

```
GOOGLE_API_KEY=AIza...your_key_here
GEMINI_MODEL=gemini-2.0-flash
```

### Step 6: Create Test Case Files

```bash
python create_test_cases.py
```

This generates all 7 input JSON files in the `test_cases/` folder.

## Design Decisions & Assumptions

### 1. LLM: Google Gemini 2.0 Flash via LangChain
`gemini-2.0-flash` was chosen for its free tier availability (1500 req/day), speed, and strong instruction-following. `gemini-1.5-flash` was deprecated and returns 404. Integrated via LangChain's `ChatGoogleGenerativeAI` for clean prompt management.

### 2. L1 Reasoning is LLM-Powered
Rather than hardcoded rules, L1 uses Gemini with a system prompt that strictly enforces visibility rules. A `VALID_L1_TARGETS` whitelist validates every planned task at parse time — any hallucinated target is remapped with a warning. Falls back to a safe default plan if all retries fail so the system never crashes.

### 3. L2 Agent Ordering Enforced in Code
An `AGENT_ORDER_PRIORITY` dict ensures agents always run in logical order regardless of what the LLM suggests — extraction before validation before tracking, Q&A before message delivery. This prevents nonsensical outputs like tracking before extraction.

### 4. Dependency-Aware Execution (Topological Sort)
The engine resolves `depends_on` relationships using topological sort before execution. A task that depends on others will only run after all its dependencies complete. Cycle detection prevents infinite loops with graceful fallback.

### 5. Structured Context Propagation
Completed task outputs are passed to downstream tasks as clearly labelled sections (e.g. `[TASK-001 | knowledge_retrieval]: ...`) so later agents can reason about what was already extracted — not just raw text.

### 6. Robust JSON Parsing with Retry
A three-strategy `extract_json()` utility handles all common LLM response issues: stripping markdown fences, extracting JSON from prose, bracket-counting for nested objects, and removing trailing commas. Retries up to 3 times with exponential backoff before using a fallback.

### 7. Graceful Ambiguity Handling
MSG-106 (null project, unknown sender, vague content) still produces a valid orchestration map — `knowledge_retrieval` attempts context lookup and `qna` generates a gap-aware response that explicitly acknowledges the missing information.

### 8. Temperature = 0.3
Low temperature ensures consistent, near-deterministic outputs across retries while still allowing LLM flexibility for content generation.
